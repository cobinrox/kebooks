Git Workflow

This chapter demonstrates the basic workflow of using Git and its basic commands. These are the common commands that you will normally be running in a multi-person team environment, with a remote Git server hosting GitHub, Bitbucket, or something similar.

There are only a handful of steps that you normally run through and are usually executed in the order summarized below

  1. git clone to download the remote repo that you've been assigned
  2. git checkout/checkout -b to get to the particular branch you've been assigned or to create a new branch for your assignment
  3. Make your local changes
  4. git add to move your changes to the staging area
  5. git commit to move your changes to the committed area
  6. git pull optionally, but best practice to download any updates that were made up on the repo
  7. git push move your changes up to the remote repo
  8. Wash rinse, repeat: continue to make local changes, stage, commit, pull, and push your branch as you develop day-to-day
  9. When your last change has been made for your work item or feature on your branch and has been pushed up to the remote, request a "merge request" on your branch, i.e. request that your branch be merged into the master (or the develop/sprint branch or whatever parent your branch is a child of). The term "merge request" is synonomous with "pull request" depending on which back-end Git server is being used. However, don't confuse "pull request" with the command "git pull". Again, "git pull" is used to download the remote repo's changes into your local repo, while a "pull request" is a part of team workflow where you request a merge of your code into the main or parent branch up on the remote.
