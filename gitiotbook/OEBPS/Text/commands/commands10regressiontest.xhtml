Regression Tesing

This is not a Git command but this section emphasizes the importance of running a regression test locally against your code repo after having pulled the parent (or any remote branch) to ensure the code still works properly.

No matter how magical Git is purported to be, it will not prevent bad things from happening. If you add a 28th ammendment to the constitution, for example, but Molly has already gotten there first and added a 28th ammendment, who wins? In the branch workflows that were discussed in the Branching chapter, everyone wins if proper testing procedures are observed. What would be the correct procedure for the 28th ammendment issue? First, Molly must make sure to provide a test that checks for 28 ammendments and that the 28th is her text ("Free ice cream"). Then, when you pull down Molly's updates into your local repo, you will be required to merge her ammendemnt with yours, creating either a 29th ammendment, or changing the text of the 28th ammendment. Which resolution to use will be based on the team's requirements. In anycase, you are then responsible for providing a test to ensure the requirements are met, i.e. that the test either checks for 29 ammendments or checks the content of the 28th ammendment.

Regression Testing: Bottomline

  • Be sure to do continuous testing of your code/project after doing a pull of the most recent parent branch.
  • This is what can prevent nasty surprises down the CI/CD pipeline.
